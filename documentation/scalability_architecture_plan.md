# Scalability & High Availability Architecture Plan (Future Scope)

This document outlines the architectural roadmap for scaling the False Alert Suppression pipeline to handle "alert storms" (e.g., hundreds or thousands of simultaneous alerts generated by DNAC). 

Currently, the pipeline processes alerts sequentially. To achieve true enterprise-grade high availability (HA) and high throughput, the following architectural upgrades should be implemented in future phases.

---

## 1. Horizontal Scaling via Containerization (Docker & K8s)

The most robust way to handle massive alert volume is to scale the consumers horizontally. Because the pipeline utilizes RabbitMQ, message distribution is handled natively via round-robin dispatch.

**Future Implementation:**
- **Containerize the Consumer:** Package `consumer-delay-queue.py` into a lightweight Docker image (e.g., using `python:3.11-slim`).
- **Orchestration (Kubernetes/Docker Swarm):** Deploy the consumer as a Kubernetes Deployment or Swarm Service. 
- **Auto-Scaling (KEDA):** Implement Kubernetes Event-driven Autoscaling (KEDA) tied to the RabbitMQ queue depth. 
  - *Trigger:* If `dnac.alerts.delayed.q` has > 50 messages, KEDA automatically spins up additional consumer pods (e.g., scaling from 2 to 20 replicas) to process the backlog in parallel.

---

## 2. Asynchronous Ingress Webhook (FastAPI)

Currently, the pipeline is initiated by manually executing `run.py`. If DNAC sends hundreds of webhooks simultaneously, a standard script executor will crash or drop packets.

**Future Implementation:**
- **FastAPI Front-Door:** Wrap the LangGraph execution (`build_graph()`) inside an asynchronous FastAPI endpoint (`POST /api/v1/alerts/ingest`).
- **ASGI Workers:** Run the API using `uvicorn main:app --workers 8`. This allows the server to ingest hundreds of concurrent HTTP POST requests instantly without blocking.
- **Background Tasks:** The FastAPI endpoint should immediately return a `202 Accepted` to DNAC, and hand the alert off to a FastAPI `BackgroundTasks` queue (or celery) for the actual LangGraph processing, ensuring DNAC never experiences a timeout.

---

## 3. Resilience and Rate Limit Protection (Exponential Backoff)

Handling high volume means external APIs (ServiceNow, OpenAI/LLMs, DNAC) will likely rate-limit our requests.

**Future Implementation:**
- **Tenacity Integration:** Wrap the API calls in `workflow/tools/servicenow_client.py` and the ML Agent 2 calls with the `tenacity` Python library.
- **Circuit Breakers:** Implement a retry mechanism with Exponential Backoff + Jitter. If ServiceNow returns an `HTTP 429 Too Many Requests`, the pipeline will safely sleep for a few seconds and retry, preventing the entire pipeline from failing during an alert storm.

---

## 4. Database Connection Pooling

As concurrency increases, opening a new MongoDB connection per alert is highly inefficient.

**Future Implementation:**
- **Persistent Client:** In `workflow/tools/mongodb_client.py`, ensure the `MongoClient` is initialized globally at the module level rather than per-request. PyMongo natively handles connection pooling (default maxPoolSize=100).
- **Bulk Upserts:** Instead of saving alerts one by one, the Reporter node can buffer results in memory and perform an `insert_many` or `bulk_write` every 5 seconds to drastically reduce I/O load on the database.

---

## 5. Message Prefetch Optimization

**Future Implementation:**
- In `consumer-delay-queue.py`, the `channel.basic_qos(prefetch_count=1)` is currently optimized for safety (processing one message at a time). 
- Once the consumer is made asynchronous or multi-threaded, the `prefetch_count` should be increased (e.g., to 20 or 50). This allows RabbitMQ to stream chunks of messages to the consumer at once, drastically reducing network round-trip latency.
