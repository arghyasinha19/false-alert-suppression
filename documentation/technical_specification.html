False Alert Suppression: Technical Specification Document

1. Introduction & Overview

The False Alert Suppression system is an intelligent, event-driven orchestration pipeline designed to ingest networking alerts from Cisco DNAC, evaluate their validity using Machine Learning and historical heuristics, and conditionally escalate them to ServiceNow.

By implementing an intelligent "delay and re-check" loop via RabbitMQ, the system eliminates transient/flapping alerts, drastically reducing manual engineering overhead and MTTR (Mean Time to Resolution).

2. System Components

2.1 LangGraph Orchestration Engine

The core pipeline uses LangGraph to maintain a strictly typed GraphState dictionary, routing alerts sequentially through specialized agents.

  • Agent 1 (Backdated Filter): Analyzes the timestamp attribute of the alert. If the event occurred significantly in the past (e.g., > 1 hour ago), the alert is flagged as is_backdated=True and gracefully suppressed.
  • Agent 2 (ML Classifier): Feeds the alert payload into a Large Language Model (configured via OpenAI API) utilizing a highly specific prompt template to classify the alert into two distinct categories: Auto resolving or Non-Auto Resolving.
  • Agent 3 (DLX Scheduler): Conditionally triggers only for Auto resolving alerts. It utilizes the pika library to serialize the alert state and push it to a Dead-Letter Exchange delay queue.
  • Agent 4 (ServiceNow Orchestrator): Conditionally triggers only for Non-Auto Resolving alerts. It interfaces with the ServiceNowClient tool to execute stateful updates against the SNOW API.

2.2 Event-Driven Delay Mechanism (RabbitMQ)

To avoid blocking compute threads during the "wait" period, the system relies on native RabbitMQ features:

  • Message TTL: Messages published to dnac.alerts.wait.q are configured with a x-message-ttl: 900000 (15 minutes).
  • Dead-Letter Routing: Upon TTL expiration, RabbitMQ natively forwards the message to dnac.exchange using the x-dead-letter-routing-key, placing it into the active consumer queue.

2.3 Consumer Daemon & DNAC Re-Check

The consumer-delay-queue.py script is a continuously running background process utilizing a blocking AMQP connection. When a delayed alert expires and hits the active queue:

  1. The consumer invokes the check_dnac_status() function to query the DNAC REST API.
  2. If DNAC reports the alert as resolved, the consumer ACKs the message, and the lifecycle terminates.
  3. If the alert remains active, the consumer dynamically injects the alert into Agent 4 by mocking the GraphState, forcefully overriding the ML prediction to Non-Auto Resolving.

2.4 Persistence & Traceability Layer

  • MongoDB: A NoSQL database initialized via MongoDBClient. The pipeline utilizes update_one(upsert=True) keyed on event_id to track the alert lifecycle across multiple asynchronous runs.
  • FastAPI Bridge: A Python web server providing the GET /api/alerts endpoint. During API execution, it extracts all unique ServiceNow Incident IDs from the MongoDB dataset and executes a bulk-fetch query (sysparm_query=numberIN...) against ServiceNow to map live incident states in real-time.
  • React Dashboard: A Vite SPA utilizing standard React hooks, Recharts, and Lucide icons. It polls the FastAPI bridge on a 15-second interval (setInterval) to render highly responsive DOM updates and analytical visualizations.

5. Security & Error Handling

  • Environment Isolation: All credentials and URIs are strictly decoupled from the codebase using the python-dotenv library. The system will fail fast if critical environment variables are missing.
  • AMQP Fallbacks: If the consumer fails to trigger ServiceNow (e.g., due to network timeouts), the try/except block catches the exception and explicitly NACKs the message with requeue=False. This prevents the RabbitMQ queue from entering an infinite loop of failures, though a final "Poison Message" queue could be implemented for enhanced observability.
