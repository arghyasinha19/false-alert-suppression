False Alert Suppression: User Manual

Welcome to the False Alert Suppression system. This manual provides Operations Teams, Engineers, and Analysts with instructions on how to access, monitor, and interact with the various components of the platform.

1. The Traceability Dashboard (Frontend UI)

The React Dashboard is your primary lens into the health and status of all networking alerts in the ecosystem.

Accessing the Dashboard

  • Local URL: http://localhost:3000 (or your server's IP).

Navigating the UI

  • KPI Hero Cards: At the top of the dashboard, you will see three glowing cards: Backdated, Auto-Resolving, and Non-Auto Resolving. These display the total lifetime volume of alerts in each category.
  • Interactive Filtering: The KPI cards are clickable! Click on "Auto-Resolving" to instantly filter the data table and charts to show only alerts that were successfully suppressed. Click it again (or refresh) to clear the filter.
  • Traceability Matrix: The large data table at the bottom shows individual alert details. The ServiceNow Status column is fetched in real-time. If it says Pending / Delayed Queue, that means the 15-minute wait timer hasn't expired yet!

2. The Data API (FastAPI Backend)

The backend serves the MongoDB data and enriches it with real-time ServiceNow data. Because it is built on FastAPI, it automatically generates interactive API documentation for developers.

Accessing the API Docs

  • Interactive Swagger UI: http://localhost:8000/docs
  • Alternative ReDoc UI: http://localhost:8000/redoc
Tip: You can navigate to /docs in your browser to interactively test the API endpoints without writing any code!

Available Endpoints

  • GET /api/alerts: Returns a JSON array of all processed alerts. The backend automatically queries ServiceNow on-the-fly, so the JSON payload will include a live_snow_status attribute for each incident.

3. Monitoring the RabbitMQ Message Broker

If you suspect alerts are not progressing, or you want to see exactly how many alerts are currently paused in the "15-minute delay" window, you should check RabbitMQ.

Accessing the Management UI

  • Local URL: http://localhost:15672
  • Default Credentials: Usually guest / guest (or check your .env file).

What to look for:

  • Navigate to the Queues tab.
  • Look at dnac.alerts.wait.q. The "Ready" column tells you exactly how many alerts are currently waiting for their 15-minute timer to expire.
  • Look at dnac.alerts.delayed.q. This should generally be 0. If messages are backing up here, it means the Python Consumer Daemon is offline or crashed.

4. Monitoring the Background Daemons

The backend Python scripts run silently. To troubleshoot or view execution traces, you must inspect their console logs.

The Consumer Daemon

The consumer orchestrates the final DNAC re-check and ServiceNow escalations. If it runs via a terminal, simply look at the stdout logs. It will clearly state:

[INFO] Checking DNAC status for device UK-LON-SW01...
[WARNING] Alert is STILL ACTIVE in DNAC after 15 minutes! Forcing Non-Auto Resolving escalation.
[INFO] Agent 4: Found active incident INC00123. Appending comment.

The Ingestion Pipeline (LangGraph)

Every time an alert is ingested via run.py, LangGraph outputs a trace of its path (e.g., Agent 1 -> Supervisor -> Agent 2) and writes a status.json file to the root directory for immediate local inspection.
